# 全局规范

总是使用中文回答。

## 开发原则

- **KISS**：追求代码和设计的极致简洁，拒绝不必要的复杂性
- **YAGNI**：仅实现当前明确所需的功能，不做过度设计
- **DRY**：杜绝重复代码，主动抽象和复用
- **SOLID**：遵循单一职责、开闭、里氏替换、接口隔离、依赖倒置原则

## 代码风格

### 通用

- 函数行数不超过 50 行，超出则拆分
- 嵌套层级不超过 3 层
- 单个文件不超过 300 行
- 避免魔法数字，使用具名常量
- 错误处理不得吞没异常

### 命名规范

- 变量/函数：camelCase
- 常量：UPPER_SNAKE_CASE
- 类/组件/类型：PascalCase
- 布尔变量：使用 is/has/should/can 前缀

### 类型系统

- 始终使用严格类型
- TypeScript 中禁止 any，使用 unknown 替代
- 函数参数和返回值必须有类型注解

### 注释

- 全部使用中文
- 复杂逻辑说明"为什么"而非"做了什么"
- TODO 格式：`// TODO(作者): 描述`
- FIXME 格式：`// FIXME(作者): 描述`

## Git 提交规范

遵循 Conventional Commits 格式，中文描述：

- `feat(scope): 新增功能`
- `fix(scope): 修复缺陷`
- `docs(scope): 文档更新`
- `refactor(scope): 代码重构`
- `chore(scope): 杂务维护`

禁止在 commit message 中使用 emoji。

## 工作流

1. 先阅读现有代码，理解架构和上下文
2. 确认需求边界，有歧义时主动提问
3. 严格遵循 SOLID/KISS/DRY/YAGNI 原则
4. 每次变更保持最小化，可独立验证
5. 代码变更后运行相关测试验证

## 安全红线

- 绝不在代码中硬编码密钥、令牌、密码
- 绝不未经确认删除文件或执行破坏性操作
- 绝不主动执行 git commit/push（除非明确要求）
